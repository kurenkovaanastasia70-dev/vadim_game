Урок: полный разбор коммита с радио, сохранениями и коллизиями

Цель урока: пройти все изменения коммита по порядку без темы контроля версий: требование, диагностика, модель состояния, предметы, UI, сохранения, коллизии и проверки.

Цели урока

Plan

Работать от чеклиста

Показать ученику, как брать задачу из плана разработки и закрывать подпункты без хаотичных правок.

Gameplay

Радио как риск

Радио должно давать полезный ответ один раз, тратить заряд и заставлять игрока покупать новое.

State

Сохранения

Таймер охоты, количество радио и позиция призрака сохраняются и восстанавливаются из `save.json`.

Collision

Физика игрока

Игрок подходит ближе к стенам, а предметы больше нельзя ставить через стену.

Шаг 1: карта изменений коммита

Учебная идея: перед кодом ученик должен понимать, какие системы затронуты и почему это один связный фикс.

`inventory_system.py`

Радио стало count-based расходником, получает cooldown, показывает ответ с таймером охоты и списывает один заряд.

`main_work.py`

Хранит таймер охоты, feedback радио, звук, сохранение охоты, призрака и новый меньший hitbox игрока.

`ghost.py`

Сериализует и восстанавливает runtime-состояние призрака: тип, позицию, состояние FSM и таймеры.

`draws.py` / `assets.py`

Рисуют overlay радио, показывают количество радио, центрируют спрайт игрока и опционально грузят звук.

Квиз: почему это один урок, а не четыре отдельных?

Потому что изменения связаны игровой петлёй: радио даёт информацию о призраке и охоте, а значит должны корректно работать заряд, таймер, сохранение состояния мира и честные коллизии взаимодействия.

Шаг 2: читаем `DEVELOPMENT_PLAN.md`

Блок 3 задаёт порядок работы:

  1. 3.1 Таймер охоты.
  2. 3.2 Ответ радио с примерным временем.
  3. 3.3 Антиспам: кулдаун или заряд.
  4. 3.4 Звук и синхрон с `inventory_system.Radio`.
  5. 3.5 Отдельная фраза во время охоты.

Отдельно пункт 5.2 требует визуал радио: короткий индикатор, шум, ответ без длинного текста.

Квиз: почему не начинаем с UI?

UI должен показывать состояние игры, а не заменять его. Если сначала нарисовать красивую рамку, но не сделать таймер и расход заряда, получится декорация без игровой логики.

Шаг 3: диагностика кода

Команды для ученика:

rg -n "radio|радио|ask_radio|hunt|item_counts" main_work.py inventory_system.py ghost.py draws.py
rg -n "save_game|load_game|serialize_hunt_state|restore_hunt_state" main_work.py

Что ищем:

  • `inventory_system.Radio.use()` — основная точка использования предмета.
  • `GhostManager.ask_radio()` — фраза призрака и проверка, есть ли у типа призрака radio-улика.
  • `Game.get_hunt_radio_text()` — текст про охоту.
  • `Game.save_game()` и `Game.load_game()` — сохранение счётчиков и таймеров.

Шаг 4: радио становится расходником

Проблема: обычный предмет покупается один раз и остаётся в инвентаре навсегда. Для радио это ломает баланс: игрок может бесконечно спрашивать призрака.

Фикс: радио добавляется в `item_counts`, как соль, кровь, крест и аккумулятор.

self.item_counts = {
    ItemType.BATTERY: 0,
    ItemType.BLOOD: 0,
    ItemType.CROSS: 0,
    ItemType.RED_DUST: 0,
    ItemType.SALT: 0,
    ItemType.RADIO: 0,
}

После использования заряд списывается:

if ok:
    game.progress_event("radio_answer", 1)
game.inventory_manager.decrease_count(self.item_type)
Квиз: почему заряд списывается после попытки, а не только после успешного ответа?

Потому что игрок всё равно использовал устройство. Если списывать только успешные ответы, можно бесконечно проверять типы без radio-улики и не платить за риск.

Шаг 5: таймер охоты

В `Game` хранится два числа:

  • `hunt_cooldown_ticks` — сколько тиков до следующей охоты.
  • `hunt_active_ticks` — сколько тиков осталось текущей охоте.
def tick_hunt_timer(self):
    if self.hunt_active_ticks > 0:
        self.hunt_active_ticks -= 1
        if self.hunt_active_ticks <= 0:
            self.reset_hunt_timer()
        return
    if self.hunt_cooldown_ticks > 0:
        self.hunt_cooldown_ticks -= 1

Важно: таймер тикает только когда игра не на паузе. Журнал, магазин и game over не должны незаметно съедать время.

Квиз: почему таймер в `Game`, а не в `Radio`?

Радио только читает состояние мира. Охота существует независимо от радио: она должна продолжаться, даже если игрок не покупал радио. Поэтому таймер принадлежит игровому состоянию.

Шаг 6: ответ радио с погрешностью

Радио не говорит точное число. Оно даёт приблизительное время, а погрешность растёт со сложностью.

error_by_difficulty = {0: 5, 1: 8, 2: 12, 3: 18}
approx = max(0, seconds + random.randint(-error, error))
return f"До охоты: ~{minutes:02d}:{rest:02d}."

Так игрок получает полезную информацию, но не превращает радио в точный секундомер.

Квиз: зачем добавлять случайную погрешность?

Погрешность сохраняет напряжение. Игрок понимает порядок времени, но не может математически идеально безопасно играть вокруг таймера.

Шаг 7: отдельная фраза во время охоты

Плохой вариант: во время охоты говорить «До охоты: ~00:00». Игрок может подумать, что охота ещё впереди.

Фикс: если `hunt_active_ticks > 0`, радио говорит отдельную фразу.

if self.hunt_active_ticks > 0:
    return "Оно здесь. Охота уже началась."
Квиз: почему это важнее красивой формулировки?

Это честная обратная связь. Игрок должен понимать текущее состояние опасности, иначе механика кажется случайной и несправедливой.

Шаг 8: звук и визуал радио

Звук сделан опциональным: если появится файл `radio_static.wav`, `radio_static.ogg`, `radio_noise.wav`, `radio_noise.ogg` или `radio.mp3`, игра его подхватит.

self.radio_static_sound = assets.load_radio_static_sound()

def trigger_radio_feedback(self, ok):
    self.radio_feedback_ok = bool(ok)
    self.radio_feedback_until = pygame.time.get_ticks() + 900
    if self.radio_static_sound:
        self.radio_static_sound.play()

Визуал — короткий overlay с полосками шума. Он не закрывает центр экрана надолго и не заменяет геймплей.

Квиз: почему звук опциональный?

В репозитории может не быть аудио-ассетов. Хорошая игровая логика не должна падать только потому, что звук ещё не добавили.

Шаг 9: сохранения

В слот сохраняются три связанные части:

  • `item_counts.RADIO` — сколько радио осталось.
  • `hunt_state` — cooldown и активная охота.
  • `ghost_state` — тип и позиция призрака при перезаходе.
save_data = {
    "item_counts": item_counts_serial,
    "hunt_state": self.serialize_hunt_state(),
    "ghost_state": self.ghost_manager.serialize_runtime_state(),
}

Для старых сейвов есть fallback: если раньше радио было boolean-предметом, `True` превращается в один заряд.

Квиз: почему нужно сохранять позицию призрака вместе с радио?

Радио сообщает информацию о мире: комнату, активность, время до охоты. Если после загрузки призрак каждый раз появляется заново, сохранение ломает честность подсказки и ощущения непрерывного раунда.

Шаг 10: нельзя ставить предметы через стену

Проблема: старая проверка разрешала ставить предмет в соседнюю клетку и проверяла, что сам предмет не пересекает стену. Но она не проверяла, есть ли стена между игроком и точкой установки.

Фикс: добавлена проверка отрезка от центра игрока до точки установки через `Rect.clipline()`.

def _placement_line_blocked(self, x, y) -> bool:
    start = self.game.player_rect.center
    end = (int(x), int(y))
    for wall_rect, _ in getattr(self.game, "walls", []):
        if wall_rect.clipline(start, end):
            return True
    return False

Та же проверка используется и для подсветки допустимых клеток, чтобы UI не обещал игроку невозможное действие.

Квиз: почему недостаточно проверить только hitbox предмета?

Потому что предмет может стоять в свободной клетке за стеной. Его hitbox не пересекает стену, но игрок физически не должен иметь возможность поставить его через препятствие.

Шаг 11: уменьшаем hitbox игрока

Проблема: `player_rect` был равен визуальному размеру спрайта. Из-за этого игрок упирался в стену слишком рано и не мог подойти близко.

Фикс: добавили отдельный визуальный размер и отдельный collision-size.

self.player_visual_size = int(TILE_SIZE * MAP_SCALE)
self.player_size = max(24, int(self.player_visual_size * 0.68))
self.player_rect = pygame.Rect(0, 0, self.player_size, self.player_size)
self.player_rect.center = (
    self.start_x + self.player_visual_size // 2,
    self.start_y + self.player_visual_size // 2,
)

В `draws.py` спрайт рисуется прежнего размера, но центрируется по меньшему `player_rect`. Поэтому визуально персонаж не «усох», а коллизии стали честнее.

Квиз: почему нельзя просто уменьшить спрайт?

Проблема была не в картинке, а в физике. Если уменьшить спрайт, игрок станет визуально меньше, но это не объясняет, где находится реальное тело персонажа. Правильнее разделить visual rect и collision rect.

Проверки после исправления

  • `python -m py_compile ghost.py main_work.py inventory_system.py draws.py mechanics.py handlers.py assets.py` проходит без ошибок.
  • После покупки радио в магазине статус показывает `Есть: 1`.
  • После использования радио счётчик уменьшается до 0, предмет исчезает из видимого инвентаря.
  • Повторное нажатие `R` без покупки нового радио показывает, что предмет не куплен.
  • При активной охоте радио пишет «Оно здесь. Охота уже началась.».
  • При обычном состоянии ответ содержит примерное время до охоты с `~`.
  • После сохранения и загрузки сохраняются заряд радио, таймер охоты и позиция призрака.
  • Если аудио-ассета нет, игра не падает и просто показывает визуальный индикатор.
  • Предметы нельзя поставить за стеной, даже если клетка рядом с игроком.
  • Игрок визуально того же размера, но подходит ближе к стенам из-за меньшего collision-rect.

Как объяснить diff ученику

  1. `inventory_system.py`: предмет радио теперь проверяет count, вызывает feedback, добавляет таймер охоты к тексту и списывает заряд.
  2. `main_work.py`: хранит radio feedback, звук, таймер охоты, сохранение `hunt_state` и `ghost_state`.
  3. `ghost.py`: умеет сериализовать и восстановить runtime-состояние призрака.
  4. `draws.py`: рисует короткий radio overlay и показывает количество радио в HUD.
  5. `assets.py`: опционально загружает звук радио.
  6. `DEVELOPMENT_PLAN.md`: чеклист обновлён после реализации, а не до неё.

Развёрнутые вопросы и ответы

Почему `Radio.use()` должен быть главным местом логики?
Потому что это объект предмета. Если часть поведения оставить в `Game.use_radio()`, а часть в `InventoryManager`, появятся два сценария с разными правилами расхода, кулдауна и прогресса задач.
Почему cooldown не списывает заряд?
Cooldown означает, что устройство ещё не готово к новой попытке. Игрок не получил новую информацию и не должен терять второй предмет из-за слишком быстрого нажатия.
Почему радио списывается даже при ответе «тишина»?
Это честная цена проверки. В расследовании отрицательный результат тоже информация: если радио молчит, игрок может исключить radio-улику в журнале.
Почему таймер охоты сохраняется?
Иначе можно было бы выходить и заходить, чтобы сбрасывать опасную фазу. Сохранение таймера закрывает такой абуз и делает раунд непрерывным.
Почему позиция призрака сохраняется отдельно от таймера?
Таймер говорит о времени, а позиция — о мире. Радио может сказать, что призрак рядом или в комнате N; после загрузки это должно остаться правдой.
Почему текст радио разделён на две строки?
Первая строка отвечает за призрака: комната, настроение, наличие radio-улики. Вторая строка отвечает за охоту. Так игрок быстрее считывает смысл.
Почему в `DEVELOPMENT_PLAN.md` надо обновлять чекбоксы?
План разработки — это рабочий документ. Если не отмечать закрытые пункты, команда будет заново обсуждать уже сделанную работу или неправильно планировать следующую итерацию.
Что будет, если в старом сейве нет `ghost_state`?
Загрузка не падает. Игра использует обычный спавн уровня. Это важно для обратной совместимости сохранений.
