5-12 мин

Bugfix - это маленькое расследование

Симптом

То, что видит игрок или разработчик: игра продолжается с 0 HP, уровень долго грузится.

Причина

Конкретное место в коде: нет перехода в новое состояние, создаётся большая unused surface.

Фикс

Минимальное изменение, которое устраняет причину и не меняет поведение соседних систем.

Регрессия

Проверка, что баг ушёл, а старые сценарии остались рабочими.

Задание на 3 минуты

теория

Как баги вообще обнаруживают

Баг редко появляется как готовая строка в списке задач. Обычно есть сигнал, который нужно превратить в воспроизводимый сценарий.

Игровое наблюдение

Разработчик или тестировщик играет и замечает странность: персонаж живёт с 0 HP, экран моргает, предмет исчезает.

Логи и метрики

FPS проседает, загрузка стала дольше, память растёт, в консоли повторяются ошибки или debug-сообщения.

Код-ревью

Баг можно найти без запуска: например, видно, что функция создаёт результат, но никто его не использует.

Регрессия

Раньше работало, после изменения сломалось. Тут важны маленькие коммиты и понятный diff.

Правило хорошего bug report

Перед исправлением нужно записать воспроизведение. Если баг нельзя повторить, его трудно проверить после фикса.

Название: при 0 HP игра не переходит в проигрыш

Шаги:
1. Запустить игру.
2. Начать новую игру.
3. Дать приведению коснуться игрока несколько раз.
4. Дождаться HP = 0.

Ожидание:
Игра показывает экран проигрыша.

Факт:
Игрок остаётся в активной игре с 0 HP.

Вопрос

Что самое важное в bug report для разработчика?

Да. Разработчику нужен воспроизводимый сценарий и понятная разница между ожиданием и фактом.
12-16 мин

Работаем в отдельной ветке

Bugfix лучше делать в отдельной ветке, чтобы можно было посмотреть diff, протестировать и не смешать изменения с другой работой.

git branch --show-current
git status --short
git switch -c bugfix_game_over_vignette

Если в рабочем дереве уже есть чужие изменения, мы их не сбрасываем. Просто не трогаем несвязанные файлы.

Вопрос

Что делать, если `git status` показывает несвязанные изменения пользователя?

Правильно: чужие изменения нельзя откатывать без прямой просьбы.

25-33 мин

Почему нужен `GAME_OVER`, а не просто флаг

В проекте уже есть `GameState`: меню, игра, магазин, настройки, сохранения. Экран проигрыша - такое же состояние интерфейса, поэтому он должен жить в той же модели.

class GameState:
    MENU = "menu"
    GAME = "game"
    SHOP = "shop"
    SETTINGS = "settings"
    DIFF = "diff"
    SAVES = "saves"
    HOWTO = "howto"
    GAME_OVER = "game_over"

Вопрос

Почему `GAME_OVER` лучше, чем отдельный `is_dead = True`?

Да. Мы не добавляем параллельную модель состояния, а расширяем уже существующую.
33-45 мин

Код фикса: переход в проигрыш

После получения урона проверяем HP. Если он стал `0`, чистим временные состояния и переводим игру в `GAME_OVER`.

def enter_game_over(self):
    self.moving = False
    self.show_save_prompt = False
    self.info_message = None
    self.info_until = 0
    self.journal_open = False
    self.journal_reset_confirm = False
    self.inventory_manager.cancel_placement()
    for key in self.keys_pressed:
        self.keys_pressed[key] = False
    self.set_state(GameState.GAME_OVER)
if self.player_hp <= 0:
    self.enter_game_over()
    draws.draw_game_over(self)
    pygame.display.flip()
    return

Важно сделать `return`, чтобы текущий кадр не продолжил отрисовку игрового состояния после перехода.

Вопрос

Верно. Если результат не используется, лучший фикс - убрать работу, а не ускорять её.
63-70 мин

Код фикса: убрать ненужную работу

При успешной загрузке уровня больше не создаём тяжёлую texture. Старую функцию оставляем безопасной, чтобы остаточные вызовы не выделяли память.

# было
self._create_vignette_texture()

# стало
self.vignette_texture = None
def _create_vignette_texture(self):
    self.vignette_texture = None
    return

Более глубокий рефакторинг можно сделать позже: удалить мёртвый код полностью или заменить его настоящим кешем экранного размера. Для узкого bugfix достаточно убрать дорогой вызов.

Какие виды тестов бывают

Smoke test

Самая короткая проверка: проект запускается, главное меню открывается, новая игра стартует. Цель - понять, что сборка не мёртвая.

Regression test

Проверяет, что старое поведение не сломалось после фикса. Например, меню, магазин и сохранения всё ещё работают.

Unit test

Проверяет маленькую функцию отдельно: нормализацию сохранения, расчёт прогресса, фильтрацию призраков по уликам.

Integration test

Проверяет связку модулей: загрузка сохранения + уровень + инвентарь + состояние игры.

Manual playtest

Человек проходит сценарий руками. Для Pygame-проекта это часто самый быстрый способ проверить UX и визуальные баги.

Performance test

Замеряет FPS, время загрузки, память. Наш баг с 10240x10240 surface относится именно сюда.

Тест-пирамида для небольшой игры

Не все тесты должны быть автоматическими. В игре есть логика, которую удобно тестировать кодом, и есть визуальное поведение, которое быстрее проверять руками.

Много:
  unit tests для чистой логики
  compile/lint checks

Средне:
  integration checks для сохранений, уровней, прогресса

Мало, но обязательно:

Для этого проекта хорошая первая автоматизация - `python -m compileall -q .` плюс маленькие unit-тесты для функций без Pygame-окна.

78-83 мин

Как читать diff после bugfix

Diff должен подтверждать границы задачи. Мы ожидаем изменения только в файлах состояния, игрового цикла, отрисовки, обработчиков и учебных материалах.

git diff --stat -- gamestate.py main_work.py handlers.py draws.py
git diff -- gamestate.py main_work.py handlers.py draws.py

Хороший сигнал

Изменений мало, они связаны с двумя багами, compile-check проходит.

Плохой сигнал

В diff внезапно появились ассеты, уровни, unrelated refactor или форматирование всего проекта.

83-88 мин

Глубже: типы багов в игровых проектах

State bug

Состояние изменилось не полностью: HP стал 0, но режим игры остался `GAME`.

Performance bug

Код делает дорогую работу, которая не нужна игроку или кадру.

Persistence bug

Данные в игре меняются, но не попадают в сохранение или некорректно восстанавливаются.

UI bug

Состояние есть, но игрок не понимает, что произошло или как продолжить.

Следующий хороший bugfix для этого проекта - сохранение размещённых предметов. Но его лучше делать отдельной веткой и отдельным уроком.
